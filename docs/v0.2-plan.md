# v0.2 plan — target loader + L1-static, end-to-end

Milestone (DESIGN.md): *Target loader (`target.yaml` parse) + one runnable
lane (L1, build invariants) end-to-end against a Leonard or bosun checkout.*

This doc pins the `target.yaml` schema and the L1 acceptance criteria so the
code doesn't drift while it's being written. Read DESIGN.md's "five
abstractions" before touching this.

## The one scope decision: L1-static, not the full seed L1

The seed lane (`seed/harness/unix-socket/lanes/L1-build-invariants.sh`) does
two different kinds of work:

1. **Static**: binary `--version` vs the source version constant(s), build
   noise, help text, baseline tests green. Pure subprocess + file reads.
2. **Live MCP**: reads `serverInfo` over the stdio/socket surface and checks
   it against the binary version. That handshake is where stdio framing and
   timeouts get fragile (the seed carries an "adaptive-recv-timeout patch"
   for exactly this), and it's the thin end of the MCP-client plumbing that
   L2 and L6 will also need.

**v0.2 ships #1 only (L1-static).** It is a legitimate end-to-end lane with
no MCP client. The live `serverInfo` drift probe is a *named follow-on*
(v0.3-adjacent): when we build it, port `seed/harness/.../mcp_sock.py`'s
handshake as the reference impl, do not reinvent the timeout handling.

Porting the seed L1 wholesale is the "it'd only take a few hours" drift
CLAUDE.md's scope rule is written to catch. Resisting it here on purpose.

## What the two real targets taught us about the schema

Checked the actual repos (2026-06-01) instead of assuming:

- **Leonard** keeps `const Version = "0.54.0"` in **two** files
  (`cmd/leonard/root.go`, `cmd/leonard-hook/root.go`), no ldflags.
- **Bosun** injects version via ldflags `-X main.version` *and* carries a
  separate `const ServerVersion = "0.2.0-alpha"` in `internal/mcp/server.go`.

So a version "source" is not one file. It is a *list of sites* that are all
supposed to agree with one expected value. The drift bug class (Leonard
bughunt-12 F001, bosun bughunt-1 F001) is precisely "these sites disagree."
The schema models the list; L1 checks agreement.

Neither target uses `internal/version/version.go` — that is Columbo's own
pattern, not a universal one. The loader must not assume it.

## The schema (pinned)

```yaml
# target.yaml — operator-supplied. Columbo does not sniff this; writing the
# threat model down by hand is the point.
name: leonard                       # required
repo: ../../leonard                 # required; abs, or relative to THIS file
module: github.com/jasondillingham/leonard   # optional; PARSE-ONLY in v0.2

baseline:                           # required
  sha: cd6d8ef                      # git short SHA at audit time (operator records)
  build: "go install ./cmd/..."     # how to produce the binaries
  test: "go test ./... -short"      # baseline test command

version:                            # required (this is what L1-static consumes)
  expected: "0.54.0"                # optional: the canonical pinned version. Omit when
                                    #   the build version is dynamic (git-describe).
  command: ["go", "run", "./cmd/leonard", "--version"]  # optional, run with cwd=repo.
                                    #   Output must contain `expected`. Prefer `go run` so it
                                    #   builds from HEAD and leaves no artifact (a plain
                                    #   `./leonard` risks running a stale binary). Skipped
                                    #   (SKIP, logged) when `command` or `expected` is absent.
  sites:                            # source files that hardcode a version literal
    - { file: cmd/leonard/root.go,      symbol: Version }   # expect defaults to version.expected
    - { file: cmd/leonard-hook/root.go, symbol: Version }
    # per-site `expect:` overrides version.expected (bosun's ServerVersion is a different value)

surfaces:                           # PARSE-ONLY in v0.2 except kind: cli is unused by L1
  - kind: cli                       # cli | mcp-stdio | mcp-unix | mcp-tcp | http | go-lib
    name: leonard
    command: ["leonard"]
  - kind: mcp-stdio
    name: leonard-mcp
    command: ["leonard-mcp"]

threat_model: |                     # PARSE-ONLY in v0.2 (DESIGN: forcing this is the point)
  Local-first toolkit. Trust boundary is the .leonard/ store and the v0.52
  config-trust file moved out of .leonard/ ...

history:                            # PARSE-ONLY in v0.2
  - { round: bughunt-12, lens: "build invariants + caps + protocol", findings: 5 }
```

### Fields-yes / consumers-no line

v0.2 writes code that **reads** only: `name`, `repo`, `baseline.{build,test}`,
`version.{expected,command,sites}`. Everything else (`module`, `baseline.sha`,
all `surfaces`, `threat_model`, `history`) is **parsed and validated for
shape, then ignored**. No handlers. This keeps the loader honest and the
fixtures complete without dragging v0.3+ work forward.

## Build sequence

1. `internal/target/target.go` — structs + `Load(path string) (*Target, error)`
   using `gopkg.in/yaml.v3`. Resolve `repo` relative to the yaml file's dir.
   Validate required fields are present; unknown fields are tolerated.
2. `examples/leonard.target.yaml` and `examples/bosun.target.yaml` — both
   written as literal YAML *before* the structs are locked. Bosun is the one
   that catches Leonard-shaped assumptions (ldflags + ServerVersion split).
3. `columbo target validate <path>` subcommand in `cmd/columbo` — loads,
   reports OK or the first validation error.
4. `internal/lanes/l1.go` — L1-static runner. Probes:
   - **P1 build**: run `baseline.build` in `repo`; expect exit 0. A
     multi-package `go build ./cmd/a ./cmd/b` is a compile-check: it builds
     all commands and discards the binaries (no artifacts left). That is the
     right shape for P1. Do NOT lean on it to produce a binary for P3 — and
     do NOT use `go install` (writes to `$PATH`, where a stale copy bites:
     the installed `~/go/bin/leonard` reported 0.52.0 while HEAD was 0.54.0).
     P3 builds its own fresh binary via `go run` instead.
   - **P2 source drift**: read each `version.sites[i].file`, extract the
     value assigned to `symbol`, check it equals the site's `expect` (or
     `version.expected` if the site has none). Any mismatch (or two sites
     that should agree disagreeing) is a FINDING.
   - **P3 binary version**: run `version.command` with cwd=`repo`; stdout
     must contain `version.expected`. Mismatch is a FINDING. SKIP (logged,
     not silently dropped) when `command` or `expected` is absent — bosun's
     git-describe version has no pinned string to assert.
   - **P4 tests green**: run `baseline.test`; non-zero is a FAIL.
   - Emit plain `PASS` / `FAIL` / `FINDING` lines + a tally. **No markdown
     writer** — the audit-format findings writer is v0.3.
5. `columbo audit l1 <target.yaml>` (or similar) wires the runner to the CLI.

## v0.2 "done"

- `columbo target validate examples/leonard.target.yaml` and the bosun one
  both pass.
- `columbo audit l1 examples/leonard.target.yaml` runs **end-to-end against
  the real Leonard checkout** and prints PASS/FAIL/FINDING for P1-P4. Not
  just "validate parses" — the lane actually builds Leonard and runs it.
- `make check` stays green.
