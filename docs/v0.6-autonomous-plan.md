# v0.6 (second half) plan — the first autonomous round

Milestone (DESIGN.md): *…; first autonomous round.* The k3s runner (first
half) is done; this is the Autonomous operator mode: "operator kicks it off,
walks away, wakes up to a PR-ready audit branch with reproducers and per-lane
writeups. No model in the loop for the boring parts." Read `docs/v0.6-plan.md`.

The whole content of this milestone is the **guardrails** DESIGN says autonomous
needs that "Driven gets for free from human attention: when to stop, when to
ask for help, when a severity call needs a human." A round runner that just
commits whatever it produces is not autonomous-safe; the guardrails are the
feature.

## `columbo audit auto <target.yaml>`

Kick-off -> round -> guardrails -> PR-ready branch -> wake-up summary. Runs the
existing round (local lanes by default; `--k3s` to use the Job runner). The
target.yaml must have a real local `repo:` (the git branch is created there);
the cluster clone targets (repo:/unused) are not auto-promotable — auto is for a
local checkout of the target.

### Next-free round number (no clobber)

Autonomous is the "round N+1 to extend coverage" cadence, so auto defaults
`--round` to the next free N, instead of v0.3's write-once-refuse. (This is the
"pick the next free N" that v0.4 deferred; autonomous is where it's needed.)

**Branch-aware, or it collides on the second run.** promote commits the round
ONLY on the `audit/bughunt-N` branch and returns to main, so main's working
tree never gains bughunt-N. Computing next-N from the working-tree `audits/`
alone would return the SAME N next time and `git checkout -b` would fail on the
existing branch. So next-N = `max(working-tree bughunt-N, existing
audit/bughunt-* branches via git for-each-ref) + 1`. The temp-repo test MUST
include a second run to catch this (a single run passes either way).

### Guardrails: block vs escalate (the core design call)

Two distinct failure modes, handled differently:

- **BLOCK (do not finalize the round)** — the audit is incomplete or unsafe to
  commit, so producing a "done" branch would lie:
  - Any lane FAILED to run (incomplete coverage — a failed lane is not "0
    findings", it's "unknown"). Reuse the v0.5 honesty: a Failed report blocks.
  - The target's git working tree is dirty (committing would sweep the
    operator's WIP into the audit branch). Require a clean tree.
  - Zero lanes produced any result at all.
  On block: write nothing to git, print the blockers, exit nonzero. The
  operator wakes to "BLOCKED: lane L6 failed — fix and re-run," not a
  misleading clean branch.

- **ESCALATE (flag for a human, but still produce the branch)** — the round is
  complete but contains calls a human must make:
  - Any UNTRIAGED finding (the probe couldn't justify a severity — DESIGN's
    "when a severity call needs a human").
  - Any HIGH/CRITICAL finding (needs human sign-off before it's acted on).
  These do NOT block the commit (the human reviews the branch); they are
  surfaced in the wake-up summary AND written into the commit message, so the
  PR opens with "REVIEW NEEDED: 2 UNTRIAGED, 1 HIGH" front and center.

`internal/autonomous/Check(reports) Verdict` is pure (blockers from Failed
lanes + empty round; escalations from severities) and unit-tested. The
dirty-tree check is a git op done in the command layer (also a blocker).

**Test the escalation RENDERING, not just the classification.** leonard's real
round is all-LOW, characterized — zero UNTRIAGED, zero HIGH/CRITICAL — so a real
leonard run only ever hits the no-escalation clean path. The "REVIEW NEEDED: 2
UNTRIAGED, 1 HIGH" commit-message text and wake-up list — the output the
guardrail exists to produce — never render against leonard. So unit-test the
rendering with synthetic HIGH+UNTRIAGED findings, asserting the commit message
and summary actually CONTAIN them (the recurring "classifier tested, rendering
not" gap).

### Promote: the PR-ready branch (local, no push)

`internal/autonomous/promote.go` (git via os/exec, in the target repo):
1. Require clean tree (blocker if not).
2. Capture the current branch.
3. `git checkout -b audit/bughunt-N`.
4. Write the round into `audits/` (the existing writer), `git add` those files.
5. `git commit` with a message including the escalation summary.
6. `git checkout -` back to the original branch (leave the operator's tree as
   found; the audit branch exists with the commit, ready to push/PR).

**No push, no PR-open** — those are outward-facing and stay the operator's
explicit step. Auto stops at "branch committed locally, ready for you."

### Wake-up summary

The thing the operator reads first: round N, lanes run + per-lane status,
finding tally by severity, the branch name, and the escalation list ("these
need you"). One screen.

### Sequencing: auto must NOT pre-write

Auto runs the round WITHOUT writing -> holds reports in memory -> Check ->
promote writes INSIDE the branch (after `checkout -b`). Do NOT reach for the
existing `--write`/`writeRound` flag: it writes to `audits/` on the current
branch before checkout, destroying the clean-tree guarantee and leaking the
files onto main. promote takes a write callback it invokes post-checkout.

## Build sequence

1. `internal/autonomous/guardrails.go` + tests: Check classifies block
   (failed/empty) vs escalate (UNTRIAGED/HIGH/CRITICAL) from synthetic reports.
2. `internal/autonomous/promote.go` + tests: against a TEMP git repo (init,
   dirty-tree blocks, clean-tree creates branch+commit, returns to original
   branch). Never touches a real repo in `make check`.
3. Next-free-round helper (extend `internal/query` or local).
4. `columbo audit auto` wiring: round -> Check -> (block: report+exit) /
   (proceed: write+promote) -> wake-up summary. `--k3s`, `--round`, `--branch`.
5. Verify: a TEMP git repo as a fake target (clean -> branch+commit; a forced
   lane-fail -> BLOCKED). Then optionally one real run against the local leonard
   checkout (round 13, local branch `audit/bughunt-13`, reversible — `git
   branch -D`), to prove the real path. Confirm with the operator before
   committing to leonard.

## v0.6 (autonomous) "done"

- `columbo audit auto <target>` runs a round, applies guardrails, and on a
  clean+complete round leaves a local `audit/bughunt-N` branch with the
  written round committed + an escalation-aware commit message, returning the
  target tree to its original branch.
- A failed lane or dirty tree BLOCKS with a clear reason and no git changes.
- Guardrails + promote are unit-tested (promote against a temp repo).
- `make check` green.

## Constraint: lane builds must not dirty the target tree

`auto` runs lanes THEN checks the clean tree, so if a lane's `baseline.build`
writes an artifact into the target repo, the tree is dirty by the time the
guard runs and `auto` BLOCKS every time. Targets must keep lane builds out of
the tree: a multi-package `go build ./cmd/a ./cmd/b` discards binaries (clean),
or use `-o /tmp/...`. A single-package `go build .` (emits `./<dir>`) or
`-o ./bin` self-dirties and makes `auto` look broken. (Found live: the tiny
test target's `go build .` left a binary and blocked correctly; leonard's
multi-package build is clean.)

## Deferred

- Push / PR-open (outward — operator's explicit step).
- Retry-on-failure, wall-clock budgets (simple stop conditions only for now).
- Autonomous severity *assignment* (we escalate UNTRIAGED to a human; we never
  auto-assign — that stays the honesty line from v0.3).
