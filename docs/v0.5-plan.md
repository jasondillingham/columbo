# v0.5 plan — columbo-mcp stdio surface (audit_status + audit_findings)

Milestone (DESIGN.md): *`columbo-mcp` stdio surface with `audit_status` and
`audit_findings`.* Read `docs/v0.4-plan.md` first.

`columbo-mcp` is for **observing**, not running probes (DESIGN: "for observing
and controlling, not running probes ... the probes themselves stay out of the
MCP surface because they're chatty and budget-heavy"). So an MCP client (Claude
in a dev session) can ask "what did the last audit find?" without opening a
shell.

## Scope (the cuts)

DESIGN lists four tools; the v0.5 milestone names only the two **read** tools.

- **In scope:** `audit_status`, `audit_findings` (read the written round).
- **Deferred:** `audit_start` (kicks a round — control, and it would put the
  chatty probe machinery behind MCP, which DESIGN warns against) and
  `audit_promote` (git ops). Both are v0.6+ control surface.
- **One target for now:** the server reads ONE audits directory (flag/cwd).
  DESIGN's `target_id` multiplexing across many targets is deferred; v0.5
  accepts the arg for forward-compat but serves the configured dir.

## Data source: the written rounds (no new store)

`audit_*` read the audit-format markdown the v0.3 writer already produces
(`bughunt-N-findings.md`). We do NOT add a `.columbo/` store yet — the
markdown IS the source of truth, and `findings.ParseRollup` already round-trips
it (built + tested in v0.3). This keeps v0.5 small and dogfoods the writer's
round-trip contract.

- `internal/query/` (the "MCP query surface" logic, testable without MCP):
  - `Rounds(dir) []int` — bughunt-N numbers present, sorted.
  - `Findings(dir, round) ([]RollupRow, error)` — `ParseRollup` of
    `bughunt-N-findings.md`; round 0 = latest. NOTE: rollup-level only
    (ID/Severity/Lane/Title/Status). Reproducers/Observed/fix-shape live in the
    per-lane detail sections ParseRollup does not read. v0.5 cut: thin-but-
    disclosed — the `audit_findings` tool description must say "rollup-level;
    full reproducers in the per-lane bughunt-N-<lane>.md files," so a client
    doesn't read the omission as "this finding has no reproducer."
  - `Status(dir, round) (Status, error)` — round number, target, total, a
    severity tally, and per-lane status, derived from the parsed rows + the
    consolidated doc's Round-status lines. This summarizes the LAST COMPLETED
    round; with `audit_start` deferred there is no live/running audit, so the
    tool description says "last completed round," not DESIGN's live-ETA
    language ("lane L4 running, ETA 14 min" is the v0.6 control-surface shape).

## The server: hand-rolled, minimal, and deliberately correct

Columbo's whole purpose is finding MCP-server bugs. Shipping a server with the
exact classes it hunts (F004 code:0, F018 unbounded read, F019 silent-drop on
parse error) would be self-refuting. So hand-roll a SMALL stdio JSON-RPC server
(no heavy SDK dep; we already understand the framing from the client) and get
those classes right ON PURPOSE:

- `internal/mcpserver/` — newline-framed JSON-RPC stdio loop:
  - `initialize` -> serverInfo {name: "columbo-mcp", version: <version.Version>}.
  - `notifications/initialized` -> no-op (no response).
  - `tools/list` -> the registered tools with input schemas.
  - `tools/call` -> dispatch to a handler; unknown tool -> error **-32602**.
  - Parse error -> a JSON-RPC error **-32700** response (NOT a silent drop).
  - Unknown method -> **-32601**. Every error code nonzero.
  - **Bounded reader**: cap line length (e.g. 1 MiB) so a no-newline flood
    can't OOM us (the F018 class). Reuse the lesson from the client's
    documented limitation.
  - Tool registry: `Register(name, schema, handler func(args) (any, error))`.
- `cmd/columbo-mcp` — replace the stub: build the registry with the two tools
  (handlers call `internal/query`), add `--audits <dir>` (default `./audits`),
  run the server loop.

## Self-audit: a discrimination test, NOT circular proof

Be honest about what running L6 against columbo-mcp proves. We implement
`-32700`/`-32601`/`-32602`/bounded-read *because those are the L6 probes*, so
L6-green against columbo-mcp is a consistency check (did we wire every case we
intended?), not proof of correctness — server and probes share an author and
thus share blind spots. The genuinely independent proof already happened: L6
caught `code:0` on **leonard-mcp**, a server we did not write to our probe list.

So frame the columbo-mcp run as a **discrimination test**: the same L6 lane
should PASS the server we built to spec AND FAIL leonard-mcp's `code:0`. The
honest claim is "L6 distinguishes the correct server from the buggy one," not
"columbo-mcp passes L6 so it's correct." STATUS must use the discrimination
framing, not proof-of-life language.

- Add `examples/columbo.target.yaml` with an `mcp-stdio` surface
  (`build: ./cmd/columbo-mcp`). Run L6 (PASS expected) and L2 (cap probes
  against read-only tools should yield clean errors, not leaks). Any FINDING is
  a real wiring slip to fix before shipping.

## Build sequence

1. `internal/query/query.go` + tests (parse a fixture audits dir: Rounds,
   Findings latest+by-N, Status tally). Pure FS + ParseRollup, no MCP.
2. `internal/mcpserver/server.go` + tests: drive it with a fake stdin/stdout
   (or in-process) — initialize, tools/list, tools/call, AND the error classes
   (truncated frame -> -32700; unknown method -> -32601; unknown tool ->
   -32602; oversized no-newline -> bounded, no OOM).
3. Wire `cmd/columbo-mcp`; `--audits`. `columbo-mcp version` still works.
4. Dogfood: drive columbo-mcp with the existing `internal/probes/mcp` client
   in a test (the client tests the server — full loop), against a fixture
   audits dir.
5. `examples/columbo.target.yaml`; run L6 (and L2) against columbo-mcp; fix any
   finding.

## v0.5 "done"

- `columbo-mcp` (pointed at an audits dir) answers `initialize`, `tools/list`
  (2 tools), `audit_status`, `audit_findings` over stdio.
- The `internal/probes/mcp` client drives it end-to-end in a test (dogfood).
- `columbo audit l6 examples/columbo.target.yaml` runs against columbo-mcp and
  is clean (or any finding is fixed) — Columbo auditing Columbo's own MCP
  surface, DESIGN's proof-of-life for this layer.
- `make check` green.

## Explicitly deferred

- `audit_start` / `audit_promote` (control surface) — v0.6+.
- `target_id` multiplexing across targets — serve one dir for now.
- `.columbo/` persistent store — the written markdown is the source of truth.
- A full bughunt-1 against ALL of Columbo (not just the MCP surface) — that is
  the v1.0 self-audit; v0.5 dogfoods only the columbo-mcp surface via L6/L2.
