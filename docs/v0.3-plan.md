# v0.3 plan — findings writer + L2 + L6 (the first real audit-format output)

Milestone (DESIGN.md): *Three lanes (L1 + L2 + L6, the cheapest three) + the
findings/ writer producing valid audit-format markdown.*

This is the version where Columbo stops printing PASS/FAIL to a terminal and
starts producing the deliverable it exists for: `audits/bughunt-N-*.md` in
the same shape Leonard and bosun use. Read `docs/v0.2-plan.md` and
`docs/v0.2-deferred.md` first; this builds directly on the v0.2 lane runner.

## Scope decisions (the cuts that keep v0.3 from sprawling)

The milestone is large. Four explicit cuts, each a deferral with a home:

1. **MCP client is stdio-only.** DESIGN lists stdio / unix-socket / TCP
   surfaces. v0.3 implements stdio only (Leonard's `leonard-mcp`, and
   Columbo's own `columbo-mcp` later, are both stdio). Unix-socket and TCP
   wait. A lane whose target has no consumable stdio surface SKIPs with a
   logged reason, exactly as L1's P3 does for dynamic versions.

2. **The writer scaffolds; it does not invent severity or prose.** The
   audit-format has two halves: machine-capturable (rollup row, captured
   reproducer, observed output, JSON-RPC error code) and human/Claude-
   authored (the severity *assessment*, the "why this severity" paragraph,
   the fix shape). v0.3 emits the first half and leaves explicit
   `TODO(operator)` markers for the second. A probe MAY declare a severity;
   if it does not, the writer writes `severity: UNTRIAGED`, never a guess.
   This is the honesty line. Columbo must not auto-author findings prose.

3. **Finding IDs are per-round sequential; cross-round stability is v0.4.**
   Leonard is on bughunt-12 with IDs accumulated across rounds; that
   ID-ledger continuity is the reconcile problem (`internal/reconcile/`,
   v0.4). v0.3 assigns F001.. in discovery order within one round and notes
   the ledger gap.

4. **One target surface, run against Leonard.** v0.3's end-to-end target is
   `examples/leonard.target.yaml` (richest stdio MCP surface). Columbo's own
   first self-audit round needs more lanes than three, so it is not the v0.3
   bar.

## The findings model

v0.2's `lanes.Result` is `{Probe, Verdict, Detail}`. v0.3 needs richer
records for the writer. Introduce a `findings.Finding`:

```go
type Finding struct {
    ID        string   // F001.. assigned by the writer, per round
    Severity  Severity // CRITICAL|HIGH|MEDIUM|LOW|UNTRIAGED
    Lane      string   // "L2 caps", "L6 protocol"
    Title     string   // one-line, goes in the rollup table
    Status    string   // confirmed|withdrawn|reference (default: confirmed)
    Files     []string // repo-relative paths into the TARGET
    Reproducer string  // runnable verbatim (captured by the probe)
    Observed  string   // what happened
    Expected  string   // the contract that was violated
    FixShape  string   // smallest change; TODO(operator) if probe can't size it
    Discovered string  // DESIGN's "date + lane" stamp, filled at write time
}
```

A probe returns `Result` for the live PASS/FAIL/SKIP tally (kept) AND, when
its verdict is FINDING, a partially-filled `Finding`. The probe fills what it
can capture mechanically (Title, Files, Reproducer, Observed, Expected); the
writer fills ID and the date; the operator fills Severity (if UNTRIAGED) and
FixShape. Keep `Result` so the terminal UX from v0.2 still works.

## internal/findings/ — the writer

Consumes `[]Finding` + round metadata, emits the audit-format set:

- `bughunt-N-brief.md` — scope, baseline (from `target.baseline`), lane
  partition, prior-round context. Mostly a scaffold with operator TODOs.
- `bughunt-N-<lane>.md` per substantive lane — the header block (Lane,
  Date, Baseline, Target, Sandbox, Reproducer-source), the severity scale
  (copied verbatim from `audits/README.md`), the rollup table for that
  lane's findings, a "Contracts re-verified (no findings)" section listing
  the PASS probes, then one detail section per finding.
- `bughunt-N-findings.md` — consolidated rollup table across all lanes (ID,
  Severity, Lane, Title, Status) + a round-status section with the severity
  tally.

The writer's contract: **the markdown it emits parses back into the same
rollup rows it was given.** That round-trip is the writer's main test (table
in, table out, IDs and severities preserved). Include at least one finding
whose Title contains a literal `|` and backticks (real titles are
pipe- and backtick-heavy, see bosun's F002/F018) so the test forces correct
table-cell escaping on the way out and un-escaping on the way back. Match the
real format from `bosun/audits/bughunt-1-findings.md` and
`bughunt-1-protocol.md` exactly: `| ID | Severity | Lane | Title | Status |`,
severity in **bold** for HIGH/CRITICAL, detail sections headed
`## F00N — SEVERITY — title`.

Output dir is a flag (default the target repo's `audits/`, since findings are
repo-relative to the target). Never overwrite an existing `bughunt-N-*.md`;
pick the next free N or require `--round N`.

### Regenerate vs hand-edit (the footgun this plan must name)

The operator fills `UNTRIAGED` severity and `FixShape` *in the emitted
markdown*. The writer also *generates* that markdown. Those collide: re-run a
lane after triage and you clobber the operator's prose. v0.3's rule is
deliberately blunt: **write-once per `(round, lane)`.** Re-running an
existing `(round, lane)` refuses unless `--force`, and `--force` overwrites
wholesale (no merge). Edit-preserving merge of operator prose into
regenerated findings is reconcile work, deferred to v0.4. The round-trip test
covers the writer in isolation; it does not cover writer-output-meets-human-
edits, and v0.3 does not pretend to.

## internal/probes/mcp — the stdio client (shared infra)

Port the handshake from `seed/harness/stdio/mcp.py` (and the
adaptive-recv-timeout lesson from `seed/harness/unix-socket/mcp_sock.py`).
Newline-framed JSON-RPC over the server's stdin/stdout:

- `initialize` -> `notifications/initialized` -> ready.
- `tools/list`, `tools/call <tool> <args>` for the clean probes.
- A `raw(frames...)` escape hatch that sends arbitrary bytes WITHOUT the
  handshake, for L6 framing fuzz that deliberately violates the contract
  (no-newline, split frame, two-in-one, BOM, batch array, deep nest).
- A per-call wall-clock timeout. The seed's hard lesson: a fixed recv
  timeout false-FAILs slow-but-correct responses. Use an adaptive deadline
  (extend while bytes are still arriving). Port this, do not reinvent it.

### Per-probe timeout vs idle grace (don't mis-report slow as silent)

The client's idle grace (~500ms after the last activity, once writes are done)
silently bounds how long a tool may take to answer with no intermediate
output: a slower tool reads as a silent-drop FINDING. Fine for Leonard's
sub-millisecond index lookups. L2/L6 must set a per-probe timeout matched to
the tool, and treat "no response within a *generous* deadline" as the
silent-drop signal, not "no response within the default grace." This is the
inverse of the F018 unbounded-read detection: tune the lane, don't false-fire.

### Launching the server — the store-dependency gotcha

`leonard-mcp` exits with `store not found... run leonard init first` if cwd
has no `.leonard/`. So the client launches the MCP server with **cwd = an
initialized target repo**, not an empty dir. v0.2 left `surfaces` parse-only;
v0.3 consumes the `mcp-stdio` surface's `command`. Two things to wire:

- Run the server fresh (`go run ./cmd/leonard-mcp`) where feasible, same
  stale-binary reasoning as L1's P3. If the surface command names an
  installed binary, document that it must be current.
- The lane needs a working dir with the target's store initialized. For
  Leonard the leonard repo itself qualifies (it is Leonard-wired). Capture
  this as a lane precondition; SKIP with a clear message if unmet.

## L2 — input caps (mostly generic probes)

From `seed/.../lanes/L2-cap-edges.sh`. The ~60% generic probes, parametrized
over the target's tools (via `tools/list`):

- Oversized string/array inputs vs documented caps.
- Integer extremes: INT64-max, INT32-max, negative, zero (the F002 class:
  servers leak Go `json: cannot unmarshal ... overflows` internals).
- Type violations: `null`, `[null]`, wrong scalar type (the F003 class:
  `reflect.Value` internals reach the operator).
- Missing/extra required fields, `additionalProperties` on edge values.

FINDING when the server leaks Go internals in an error, or silently accepts
an input it documents as invalid. These probes live in
`internal/probes/caps/` and are target-agnostic.

## L6 — protocol fuzz (framing + JSON-RPC spec compliance)

From `seed/.../lanes/L6-protocol.sh`. Two probe families:

- **Framing edges** (use the `raw` escape hatch): no-newline unbounded read
  (the F018 OOM lever), split frame across sends, two frames in one send,
  UTF-8 BOM prefix (F021), deeply nested JSON (F023), JSON-RPC batch array
  (F025), duplicate keys (F022). FINDING on silent-drop-without-error or
  connection teardown without a `-32700`.
- **JSON-RPC spec compliance** (use the clean client): error codes must be
  nonzero (F004 `code:0` class), `tools/call` before `initialized` rejected,
  second `initialize` rejected or no-op, duplicate in-flight ids, unknown
  tool -> `-32602`. FINDING on spec deviation.

These live in `internal/probes/protocol/`. Note: the seed found most of
these as LOW, and many are "exact shape as Leonard's FNNN" -- the portable-
class observation is itself worth capturing in the finding Title.

## L1 gets its live half back (rides along on the MCP client)

The deferred live-`serverInfo` drift probe (`docs/v0.2-deferred.md`) needs
exactly this stdio client. Add it as L1 P5: read `serverInfo.version` over
the wire, compare to `version.expected`. SKIP when no mcp surface. This does
NOT close the separate ldflags-vs-`go run` P3 gap (still build-to-tempdir,
still deferred, still required before the Columbo self-audit).

## Build sequence

The deliverable is the writer, and it needs zero MCP work: L1 already
produces a real FINDING (the version-drift case), so the writer can hit its
full acceptance bar fed by L1 alone. Build and *prove* it first, behind a
hard gate, before touching the riskiest piece (the stdio client: framing,
adaptive timeout, `go run` compile-delay, store-dependency cwd). Then L2/L6
are purely additive — same instinct that let v0.2 ship by cutting L1 down to
L1-static.

1. `internal/findings/finding.go` — the `Finding` type + `Severity`.
2. `internal/findings/writer.go` — emit the three doc types; round-trip test
   (rollup table in -> markdown -> parse -> same rows, pipe/backtick title
   included).
3. **GATE: writer proven against L1-only output.** Wire `columbo audit l1`
   to feed its FINDING into the writer and emit a valid `bughunt-N-*.md` set
   for Leonard. This is the milestone's named deliverable, reached with no
   MCP client. Do not start step 4 until this is green.
4. `internal/probes/mcp/client.go` — stdio JSON-RPC client + `raw`, adaptive
   timeout. Unit-test the framing/timeout against a tiny fake server.
5. `internal/probes/caps/` + `internal/lanes/l2.go`.
6. `internal/probes/protocol/` + `internal/lanes/l6.go`.
7. L1 P5 live serverInfo probe (rides on the step-4 client).
8. `columbo audit l2|l6 <target.yaml>`; `columbo audit round <target.yaml>`
   runs L1+L2+L6 and invokes the writer.

## v0.3 "done"

Two gates, in order. The first is the milestone; the second is the stretch.

**Gate A (the deliverable) — writer, proven on L1 alone:**
- The emitted markdown round-trips: its rollup table re-parses to the same
  findings, severities and IDs intact, including a pipe/backtick-laden Title.
- `columbo audit l1 examples/leonard.target.yaml` can feed its FINDING into
  the writer and emit a valid `bughunt-N-*.md` set.
- Write-once per `(round, lane)` is enforced (`--force` to overwrite).
- Severities the probe can't justify are `UNTRIAGED`, never guessed.

**Gate B (the three lanes):**
- `columbo audit round examples/leonard.target.yaml` runs L1+L2+L6 against
  the real Leonard stdio MCP server and writes a full `bughunt-1-*.md` set.
- Every FINDING carries a captured reproducer and Files.
- `make check` stays green, including the writer round-trip test and the MCP
  client framing test.

If Gate B's MCP work overruns, Gate A still ships the milestone's named
deliverable. That is the point of the ordering.

## Shipped-but-partial: L2 silent-accept (deferred)

L2 as built (Gate B step 2) detects only the leak/panic half of its planned
scope. The plan's other half — FINDING when the server "silently accepts an
input it documents as invalid" — is deferred: it needs per-field intent the
schema does not carry. Concretely, L2's PASS bucket currently cannot tell
apart (a) correctly rejected per a documented cap, (b) accepted because no cap
is documented, and (c) accepted but a cap *should* exist. Verified against
Leonard: `find_symbol`/`verify_symbol` enforce their documented 4096-byte cap
(good), and the tools that accepted a 1 MiB string document no cap (so not a
finding). No hidden finding today, but the judgment is absent. Picking it up
needs the schema's `description` caps parsed, or operator-declared expected
caps in `target.yaml`. Tracked here so the next session sees L2 is ~half its
planned scope, not whole.

## L6 verification + grader-scope caveats

- **The silent-drop FINDING path is verified only synthetically in v0.3.** L6
  exists largely to catch the F019/F021/F023/F025 silent-drop class, but the
  one stdio target available (Leonard) handles all four frames correctly
  (real `-32700` / `-32600`), so against a real server only the PASS side
  fires. bosun is the target that exhibits silent drops, but `bosun mcp` is
  unix-socket-only (v0.4), so it can't be driven here. The silent-drop branch
  is covered by `classifyL6`'s synthetic-session test, not by a real run.
  When unix-socket support lands in v0.4, re-run L6 against bosun to exercise
  it end-to-end.
- **L6's catch surface is silence + zero-code, not full spec compliance.** A
  server that rejects a batch with one `-32600`, or returns a non-error result
  for an unknown tool, reads as PASS (it didn't go silent and didn't use
  code:0). That is correct for this scope but is NOT "supports JSON-RPC §6
  batch" or "validates the tool name." Deeper compliance grading is future
  work; don't over-read the PASSes. Same shape as the L2 silent-accept cut.
