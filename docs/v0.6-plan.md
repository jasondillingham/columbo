# v0.6 plan — k3s job-based lane runner (live on the homelab cluster)

Milestone (DESIGN.md): *k3s job-based lane runner; first autonomous round.*
Operator chose **full live k3s on the homelab** (macmini + thor). Read
`docs/v0.5-plan.md` first.

This is the first milestone that leaves "local, Leonard-verifiable" and runs on
real infrastructure. The bar: `columbo audit round --k3s <target>` creates a
k8s Job per lane on the cluster, the pods probe the target, the orchestrator
collects their findings and writes a round — verified by an actual cluster run,
not just `make check`.

## Cluster facts (probed 2026-06-01)

- two nodes (control-plane + worker) on a private LAN, both **x86_64**, k3s
  v1.35.5, containerd runtime. kubectl works from here.
- **No registry, no shared PV** (the homelab's open storage TODO). Two
  consequences that shape the design:
  - **Image distribution:** `docker build` (linux/amd64) -> `docker save` ->
    `ssh <node> 'sudo k3s ctr images import -'` on BOTH nodes, Jobs use
    `imagePullPolicy: Never`. No registry needed.
  - **Findings transport:** NOT a shared volume. Each lane pod prints its
    findings as a sentinel-delimited JSON block to stdout; the orchestrator
    reads `kubectl logs` and extracts between sentinels. Sidesteps the missing
    shared-storage entirely.

## The ID-collision merger is obviated (honest scope correction)

DESIGN names `internal/reconcile/` "ID-collision merger for parallel agents,"
and earlier notes said v0.6 is where it earns its keep. Building it RIGHT shows
it isn't needed: the seed had collisions because *manual* agent sessions each
self-numbered `F0NN` into separate files. An engineered runner has lane pods
emit **ID-less** `[]findings.Finding` JSON; the orchestrator assigns F001..
centrally (the existing `Round.AssignIDs`) after collecting all lanes. No
collision, no merger. We reuse v0.4's `reconcile.Dedup` for the dup-collapse,
which is the reconciliation that's actually valuable. Net: v0.6 adds the k3s
runner, not a new merger. (The merger stays unbuilt — good outcome, recorded.)

## Pieces to build

### 1. Lane entrypoint: `columbo lane <L> <target.yaml>` (the pod command)

Runs ONE lane and prints ID-less findings as a sentinel JSON block:

```
---COLUMBO-FINDINGS-BEGIN---
{"lane":"L6 protocol","slug":"protocol","findings":[...],"reverified":[...],"skipped":[...],"failed":[...]}
---COLUMBO-FINDINGS-END---
```

All human/log output goes to stderr so stdout's block parses cleanly. Locally
testable: `columbo lane l6 examples/leonard.target.yaml` should emit the same
findings L6 produces today, just serialized. This is the unit-testable core.

### 2. In-pod target access: clone + setup

A cluster pod has no local checkout. Add an optional `clone` block to
target.yaml:

```yaml
clone:
  url: https://github.com/jasondillingham/leonard
  setup: ["go build -o bin/ ./cmd/leonard ./cmd/leonard-mcp", "./bin/leonard init ."]
```

`columbo lane`, given `--workdir /work` and a target with `clone`, does:
git clone url -> checkout baseline.sha -> run each `setup` cmd (this is where
`leonard init .` creates the `.leonard/` store the server needs) -> run the
lane with repo=/work/<clone>. Local runs (no `--workdir`) ignore `clone` and
use `repo:` as today. Leonard is public, so the pod can clone it.

### 3. Two images (slice 1 tiny, slice 2 full)

- **`columbo:slim` (slice 1):** static `CGO_ENABLED=0` build of `columbo` +
  `columbo-mcp` on `scratch`/distroless (~20 MB). No toolchain. Used for the
  first cluster slice, whose target's mcp-stdio surface is `columbo-mcp`
  itself (already on PATH in the image) — no clone, no build, no `leonard
  init`, no network egress. The 20 MB distribution loop is the one to iterate
  on while getting import/scheduling right.
- **`columbo:build` (slice 2):** `golang:1.23` + git + the columbo binary, for
  the clone/build/probe path against leonard. Bigger (~1 GB); only needed once
  the runner is already proven.

Both built from the local source tree (columbo isn't in git; `docker build`
uses the local context).

### 5. Job lifecycle (get this right from the start)

`kubectl wait --for=condition=complete` HANGS until timeout on a crashed pod.
The orchestrator must: set `backoffLimit: 0` (a failed pod must not retry and
muddy which attempt's logs we read) and `restartPolicy: Never`; wait for
complete OR failed (poll both, or wait-with-timeout then check status); and on
failure surface the pod logs rather than hang. Pods fail constantly while the
image is being gotten right, so this is the path that gets exercised most.

### 4. Job manifest + orchestrator: `audit round --k3s`

- `internal/orchestrator/k3s.go` (or a new `internal/k3srunner/`): render a
  Job per lane (name, image, `columbo lane <L> ...` command, nodeSelector to
  spread across macmini/thor, `imagePullPolicy: Never`, `restartPolicy: Never`,
  TTL cleanup). Shell out to `kubectl` (apply / wait / logs / delete) — no
  client-go dependency for v0.6.
- The orchestrator: apply Jobs -> `kubectl wait --for=condition=complete` ->
  `kubectl logs` per Job -> extract the sentinel block -> unmarshal to
  LaneReport -> assemble Round (central AssignIDs) -> Dedup -> write. Same
  writer as local.
- `--k3s` flag on `audit round`; without it, the existing local goroutine path.

## Build + verify sequence (test one axis at a time)

Each step changes ONE variable from the last, so a failure localizes.

1. `columbo lane` entrypoint + sentinel serialization. Unit-test round-trip;
   locally `columbo lane l6 examples/columbo.target.yaml` emits a block that
   parses to L6's findings (6 PASS, 0 findings). **Gate before any cluster work.**
2. **`columbo version` Job** — the dumbest pod. Build `columbo:slim`, import to
   both nodes, `kubectl apply` a Job that runs `columbo version`. Isolates "can
   the cluster pull-never-run + schedule my image" from anything lane-related.
   If this fails, it's distribution/scheduling, full stop.
3. **Slice 1 — known-good cluster lane.** A target whose mcp-stdio surface is
   `columbo-mcp` (in the slim image, no clone). Hand-run an L6 Job; `kubectl
   logs` must yield the sentinel block with **6 PASS, 0 findings** — a known
   answer, so any deviation is a pipeline bug (no SHA-matching needed). Proves
   the cluster-only parts: import, `imagePullPolicy:Never` scheduling, Job
   lifecycle (incl. the failure branch), sentinel-over-logs.
4. Wire `audit round --k3s` (Job render + apply + wait-complete-or-failed +
   logs + collect + write). Verify against slice 1: a written round, 0 findings.
5. **Slice 2 — leonard via clone.** `columbo:build` image; `clone`/`setup` in
   `leonard-cluster.target.yaml`; cluster-clone SHA == the local baseline SHA
   (or the set-identical gate is meaningless). First verify cluster egress to
   github + the Go proxy and that `leonard init` works in a fresh clone. Then a
   real L2+L6 round whose findings are set-identical (modulo IDs) to a local
   run. **Done = a real cluster round against leonard.**

(Slice 2's clone/build/init path, step 2 from the old ordering, is built here
but tested locally against a temp workdir first, then in-pod.)

## v0.6 "done"

- `columbo audit round --k3s examples/leonard-cluster.target.yaml` creates L2
  and L6 Jobs on macmini/thor, they clone+build+probe leonard, the orchestrator
  collects their findings over logs and writes a `bughunt-N-*.md` round whose
  findings match a local run (set-identical, modulo IDs).
- The image, import steps, and a hand-run Job are reproducible (documented).
- `make check` green for the local-testable parts (lane entrypoint,
  serialization, clone/setup, manifest rendering).
